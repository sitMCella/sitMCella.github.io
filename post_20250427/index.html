Server-Sent Events in React

Apr 27, 2025

Server-Sent Events (SSE) is a unidirectional, event-based communication protocol that allows a server to send data to a client over HTTP using a persistent connection. WebSocket is another technology that establishes a communication stream between the server and the client. Unlike Server-Sent Events, WebSocket enables bidirectional communication and is not based on the HTTP protocol.

The HTML specification includes support for Server-Sent Events and introduces the EventSource API, which is natively supported by most modern browsers. The EventSource API requires the server to send responses using the MIME type text/event-stream, and the messages are streamed as UTF-8 encoded text data. To open a connection with the server, you just need to create an EventSource object and then listen for the message events.

const eventSource = new EventSource('/api/devices')

eventSource.onmessage = (event) => {
  const device = JSON.parse(event.data)
  if (!devices.some((item) => device.id === item.id)) {
    setDevices((prevDevices) => [...prevDevices, device])
  }
}

eventSource.onerror = (err) => {
  console.error('Devices SSE error:', err)
}

I used the EventSource API to process a stream of data from a React web application, the GitHub repository of the project is the following:
sitMCella/radar

The web application uses React Three Fiber [1] to display a radar simulation within a canvas, rendering a grid with points that represent the positions of a set of devices and the nearby objects they detect. The main goal of the web application is to display the positions of the detected objects in near real-time. In order to achieve this goal, I developed a reactive backend, subscribed to the data stream using EventSource, and processed the received messages in the React application.

The initial version of the application handled EventSource messages on the browserâ€™s main thread. I noticed that the user experience degraded as an increasing number of messages were processed from the EventSource in short time intervals. With Server-Sent Events the client has not way to notify the server to slow down, so in order to handle the pressure from the server, I created a Web Worker to handle the EventSource messages in a separate thread.

const eventSource = new EventSource('/api/devices')

eventSource.onmessage = (event) => {
  postMessage(event.data)
}

eventSource.onerror = (err) => {
  postMessage({ error: 'SSE error', details: err })
}

In addition to the Web Worker, I also used the RxJS library to handle the asynchronous data stream, particularly because it provides a set of operators for rate limiting and buffering. The Web Worker is initialized within the RxJS Observable, and the messages it sends are forwarded to the RxJS subscriber. I have instrumented the Observable with the bufferTime [2] operator, used to batch the data emissions from a possible high-frequency observable. The bufferTime operator accumulates values from the source observable in an array over a specified time duration before emitting the buffered array.

const [devices, setDevices] = useState([])

const createDevicesObservable = () => {
  return new Observable((subscriber) => {
    const worker = createDevicesWorker()

    worker.onmessage = (w) => {
      if (w.data?.error) {
        subscriber.error(w.data)
        return
      }
      subscriber.next(w.data)
    }

    return () => {
      worker.terminate()
    }
  })
}

useEffect(() => {
  const devicesObservable = createDevicesObservable().pipe(
    bufferTime(200)
)

  let updatedDevices = []

  const devicesSubscription = devicesObservable.subscribe({
    next: (batch) => {
      batch.forEach((data) => {
        const device = JSON.parse(data)
        if (!updatedDevices.some((item) => device.id === item.id)) {
          updatedDevices = [...updatedDevices, device]
        }
      })
      setDevices(updatedDevices)
    },
    error: (err) => console.error('Devices SSE error:', err),
  })
}, [])

To update the React UI at a sustainable rate, it's important to call setState only at the end of each batch execution. Additionally, the React Hook useMemo [3] helps to ensure that the React UI re-renders the components only when the useState array is actually modified.

const renderedDevices = useMemo(() => {
  return devices.map((device, index) => (
    <Device key={index} device={device} />
  ))
}, [devices])

return (
  <div>
    {renderedDevices}
  </div>
)

The new version of the web application is significantly more responsive, and React Three Fiber accurately renders the positions of devices and objects within the canvas in near real-time, even under heavy server load.
